# Team Context Guardrails

이 문서는 팀 컨텍스트에 수집해도 되는 것과 안 되는 것을 정의합니다.
에이전트는 문서를 추가/수정하기 전에 반드시 이 가드레일을 확인해야 합니다.

---

## 절대 수집 금지

다음 내용이 포함된 경우 수집을 중단하고 사용자에게 알려야 합니다.

### 시크릿 & 크레덴셜
- API 키, 토큰, 비밀번호, `.env` 값, 인증서
- 내부 URL 중 인증이 필요한 엔드포인트의 실제 접속 정보

### 개인 정보 (PII)
- 실명 + 연락처 조합, 주민번호, 계좌번호
- 고객/사용자의 실제 데이터 (이메일, 전화번호, 결제 정보)
- 팀원의 연봉, 인사 평가, 근태 기록

### 개인적인 내용
- 특정 개인의 업무 습관이나 성향에 대한 평가
- "누구는 항상 ~한다", "누구 때문에 ~했다" 같은 표현
- 개인 간 갈등, 불만, 핀잔, 비난
- 1on1 미팅 내용, 개인 피드백
- 특정인을 지목한 실수/장애 원인 귀책

### 조직 민감 정보
- 퇴사/이동 예정 정보
- 비공개 인사 결정
- 팀 간 정치적 맥락

### 라이선스 & 저작권
- 유료 서비스의 전문 복붙
- 명시적 허가 없는 타팀/타사 내부 문서

## 수집 전 확인 필요

다음 항목은 반드시 사용자에게 확인 후 수집합니다.

| 항목 | 확인할 것 |
|------|-----------|
| 팀 합의 여부 | 한 사람의 의견인가, 팀이 합의한 것인가? |
| 다른 팀의 정보 | 우리 팀 관점의 인터페이스인가, 남의 내부 사정인가? |
| 실제 데이터 | 마스킹/익명화 되었는가? |
| 유효 기간 | 시한부 정보라면 `expires` 표기했는가? |
| 비공개 의사결정 | 아직 공유 전인 결정 사항이 아닌가? |

## 수집 권장

다음은 팀 컨텍스트에 적극 수집하세요.

- 팀이 합의한 코딩 컨벤션, 네이밍 규칙, 브랜치 전략
- 시스템/서비스 구조와 의존성
- 팀 고유 용어, 약어, 내부 시스템명 해설
- 의사결정 기록 (ADR) — 무엇을 왜 그렇게 결정했는지
- 온보딩에 필요한 지식 — 새 팀원이 첫 주에 알아야 할 것
- 자주 발생하는 문제와 해결 패턴
- 배포/테스트/리뷰 프로세스

## 올바른 표현 기준

팀 컨텍스트는 **팀 전체가 공유하는 문서**입니다.
누군가 읽었을 때 불쾌하거나 곤란할 수 있는 표현은 들어가면 안 됩니다.

| 안 되는 표현 | 괜찮은 표현 |
|-------------|------------|
| "철수가 로깅을 빼먹어서 장애 남" | "로깅 누락으로 장애 발생 → 체크리스트에 로깅 항목 추가" |
| "영희는 리뷰를 항상 늦게 함" | "리뷰 SLA: PR 생성 후 24시간 이내" |
| "이 코드 누가 짠 건지 모르겠지만 엉망" | "레거시 코드 개선 필요 → 리팩터링 계획 작성" |
| "A팀이 API를 이상하게 만들어서" | "A팀 API 연동 시 주의사항: 응답 형식이 X인 경우 처리 필요" |

**원칙: 사람이 아니라 시스템/프로세스를 개선하는 방향으로 기록한다.**

## frontmatter 권장 필드

문서 추가 시 다음 frontmatter를 포함하세요:

```yaml
---
id: api-naming              # 문서 식별자
summary: REST API 네이밍 컨벤션  # 한 줄 설명
source: 팀 회의 2026-03-10  # 출처 (회의, 문서, PR 등)
status: approved            # draft | approved | deprecated
expires: null               # 만료일 (없으면 영구)
---
```

- `status: draft` → 에이전트가 참조 시 "아직 확정되지 않은 내용"임을 명시
- `status: deprecated` → 참조하지 않음
- `expires` 지난 문서 → 사용자에게 갱신 필요 알림
